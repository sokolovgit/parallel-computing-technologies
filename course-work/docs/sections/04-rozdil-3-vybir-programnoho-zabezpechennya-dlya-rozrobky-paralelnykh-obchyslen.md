### 3.1 Характеристика обраної технології

Python є мовою високого рівня з розвиненою стандартною бібліотекою, у якій уже наявні засоби для побудови багатопотокових і багатопроцесних програм. Для цієї курсової роботи це важливо з двох причин. По-перше, реалізація алгоритму бітонічного сортування потребує багаторазового відтворення однакових compare-swap операцій, тому код повинен бути компактним і достатньо прозорим для аналізу. По-друге, експериментальна частина роботи вимагає не лише реалізації алгоритму, а й тестування, вимірювання часу виконання та варіювання параметрів паралельного запуску.

Попри те, що Python поступається мовам нижчого рівня за максимальною швидкодією, він добре підходить для прототипування та дослідження паралельних алгоритмів. У межах цієї роботи це дозволяє зосередитися на правильному поділі задачі, синхронізації процесів і спільному доступі до даних, не ускладнюючи реалізацію низькорівневими деталями керування пам’яттю.

### 3.2 Засоби для організації паралельних обчислень

Основним засобом паралельної реалізації в роботі обрано модуль `multiprocessing`. Його головна перевага полягає в тому, що він використовує окремі процеси, а не потоки, і тому дає змогу обійти обмеження `Global Interpreter Lock` у CPython. Для задачі бітонічного сортування це принципово, оскільки сортування є CPU-bound алгоритмом, де виграш очікується саме від одночасного виконання обчислень на кількох ядрах процесора.

Важливим доповненням до `multiprocessing` є модуль `multiprocessing.shared_memory`, який надає засоби спільного доступу до даних між процесами. Для паралельних обчислень це корисно тоді, коли потрібно працювати з великими масивами даних і водночас зменшити накладні витрати на їх передавання між процесами. Саме тому цей модуль доцільно розглядати як природне доповнення до процесної моделі паралелізації в Python.

Поряд з `multiprocessing` у Python існує модуль `threading`, який забезпечує запуск кількох потоків у межах одного процесу. Його перевагою є спільний адресний простір і простіша модель обміну даними. Проте для CPU-bound задач у стандартній реалізації CPython використання потоків має суттєве обмеження через `GIL`: одночасне виконання байткоду на кількох ядрах фактично не досягається. Тому `threading` доцільніше застосовувати для I/O-bound задач, а не для інтенсивного сортування великих масивів.

Як додаткову альтернативу варто розглянути модуль `concurrent.futures`. Він надає високорівневий інтерфейс `ThreadPoolExecutor` і `ProcessPoolExecutor` для запуску задач у потоках або процесах. Перевага цього підходу полягає в простішому програмному інтерфейсі й уніфікованій роботі з пулом виконавців. Проте в межах цієї роботи важливо не лише запускати паралельні задачі, а й мати достатньо явний контроль над процесною моделлю, синхронізацією та організацією спільного доступу до даних. Тому `concurrent.futures` доцільно розглядати як корисну альтернативу, але не як основний інструмент роботи.

### 3.3 Обґрунтування вибору для цієї задачі

Алгоритм бітонічного сортування має регулярну структуру, у якій значна частина операцій може виконуватися незалежно. Разом з тим між окремими етапами обчислень потрібне узгодження, щоб зберігалася правильна послідовність перетворень даних. Отже, для цієї задачі потрібен інструмент, який дозволяє використовувати кілька ядер процесора, ділити роботу на підзадачі та підтримувати контрольований обмін даними між виконавцями.

Саме цим вимогам найкраще відповідає зв’язка `multiprocessing` + `multiprocessing.shared_memory`. Вона поєднує справжній процесний паралелізм із можливістю організувати спільний доступ до даних без переходу до зовнішніх платформ або спеціалізованих фреймворків. Для навчальної курсової роботи це є важливою перевагою, оскільки обрані засоби входять до стандартної бібліотеки Python, добре документовані та достатні для побудови й дослідження паралельного алгоритму на багатоядерному CPU.

Отже, для даної курсової роботи вибір Python як мови реалізації, `multiprocessing` як основного механізму паралелізації та `multiprocessing.shared_memory` як засобу спільного доступу до даних є обґрунтованим. Модулі `threading` і `concurrent.futures` розглядаються як корисні альтернативи, однак для CPU-bound сортування великих масивів вони або поступаються за практичною придатністю, або менш точно відповідають вимогам задачі.
