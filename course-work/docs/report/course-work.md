# ВСТУП

Головним завданням курсової роботи є паралельна реалізація алгоритму
бітонічного сортування мовою програмування Python, яка забезпечує
прискорення не менше 1,2 при достатньо великій складності обчислень.

Сучасні обчислювальні системи дедалі ширше використовують багатоядерні
та багатопроцесорні архітектури, тому розробка ефективних паралельних
алгоритмів є важливою складовою високопродуктивних обчислень. Паралельне
виконання дає змогу скоротити час обробки великих обсягів даних за
рахунок одночасного виконання незалежних або частково незалежних
підзадач, однак ефективність такого підходу безпосередньо залежить від
структури алгоритму, способу розподілу роботи та накладних витрат на
синхронізацію \[1\].

Однією з базових задач, для яких активно досліджуються паралельні
методи, є сортування даних. Сортування використовується як самостійна
операція, а також як складова більш складних алгоритмів обробки, аналізу
та впорядкування інформації. Для паралельної реалізації особливий
інтерес становлять алгоритми, структура яких заздалегідь визначає
послідовність порівнянь і перестановок елементів. До таких алгоритмів
належить бітонічне сортування, запропоноване К. Бетчером у межах підходу
сортувальних мереж \[2\].

Алгоритм бітонічного сортування базується на побудові бітонічних
послідовностей та їх подальшому злитті за фіксованою схемою
компараторів. Завдяки регулярній структурі та чітко визначеним етапам
порівняння цей алгоритм є зручним для паралельного виконання, оскільки
значна частина операцій одного етапу може виконуватися одночасно \[2, 3,
4\]. Саме тому бітонічне сортування є зручним прикладом для дослідження
особливостей побудови паралельного алгоритму, організації синхронізації
та оцінювання досягнутого прискорення.

У межах цієї курсової роботи буде розроблено послідовну та паралельну
реалізації алгоритму бітонічного сортування мовою Python. Послідовна
версія алгоритму використовуватиметься як базовий варіант для перевірки
коректності результатів і для подальшого порівняння швидкодії.
Паралельна версія буде побудована на основі тих самих операцій
порівняння та обміну елементів, але з розподілом індексів між кількома
процесами та синхронізацією виконання на окремих етапах сортування.

У роботі буде проведено тестування обох реалізацій, досліджено їхню
швидкодію на масивах різного розміру, а також проаналізовано вплив
кількості процесів на отримане прискорення. Очікуваним результатом є
підтвердження доцільності паралельної реалізації алгоритму бітонічного
сортування для великих обсягів даних і визначення параметрів запуску, за
яких досягається найбільша ефективність.

# 1 ОПИС АЛГОРИТМУ ТА ЙОГО ВІДОМИХ ПАРАЛЕЛЬНИХ РЕАЛІЗАЦІЙ

## 1.1 Загальна характеристика алгоритму бітонічного сортування

Бітонічне сортування належить до алгоритмів сортування порівняннями та є
класичним прикладом сортувальної мережі (рис. 1.1). Алгоритм був
запропонований Кеннетом Бетчером \[2\]. Його ідея полягає в тому, що
довільну послідовність можна впорядковувати через побудову бітонічних
послідовностей і їх подальше бітонічне злиття.

Бітонічною називають послідовність, яка спочатку монотонно зростає, а
потім монотонно спадає, або може бути отримана циклічним зсувом із такої
послідовності \[3, 4\]. Для двох елементів така послідовність
утворюється тривіально, а далі з малих бітонічних підпослідовностей
послідовно формуються більші.

Особливість алгоритму полягає в тому, що послідовність компараторів
задається наперед і залежить лише від розміру масиву. У термінах Кнута
це відповідає oblivious sequence of comparisons, де структура порівнянь
є фіксованою, а окремі компараторні модулі можна подати як вузли мережі
\[2, 5\].

![](media/image2.png){width="6.684584426946632in"
height="3.2347889326334207in"}

Рисунок 1.1 -- Сортувальна мережа бітонічного сортування для 8 елементів

У класичному варіанті алгоритм працює для кількості елементів, що
дорівнює степеню двійки. Для масивів іншої довжини на практиці
застосовують доповнення до найближчого значення вигляду 2\^k.

Класичне подання bitonic sort часто є рекурсивним: дві половини масиву
впорядковуються у протилежних напрямках, утворюючи бітонічну
послідовність, після чого виконується бітонічне злиття \[2, 3, 4\]. Для
практичної реалізації зручнішою є ітеративна форма, оскільки вона подає
алгоритм як послідовність шарів compare-swap операцій. У такому записі
використовуються параметри k і j, а партнер для індексу i визначається
як i XOR j. Саме цей варіант краще підходить для програмної реалізації,
побудови мережі та розподілу роботи між паралельними виконавцями.

Асимптотична складність бітонічного сортування в послідовному виконанні
становить O(n log\^2 n) \[3, 4\]. Воно поступається оптимальним
послідовним алгоритмам сортування, але компенсує це регулярною
структурою, передбачуваним шаблоном доступу до даних і зручністю
розпаралелювання. На кожному етапі значна частина compare-swap операцій
є незалежною, тому вони можуть виконуватися одночасно, а синхронізація
потрібна лише між окремими шарами мережі \[1, 3, 4\].

## 1.2 Псевдокод алгоритму

+-----------------------------------------------------------------------+
| **Algorithm** Iterative bitonic sorting                               |
+=======================================================================+
| **Input**: A - array of elements, n - number of elements, where n is  |
| a power of 2.                                                         |
|                                                                       |
| **Output**: array A sorted in ascending order.                        |
|                                                                       |
| 1\. For each block size k = 2, 4, \..., n:                            |
|                                                                       |
| 2\. For each stride j = k / 2, k / 4, \..., 1:                        |
|                                                                       |
| 3\. For each index i = 0, \..., n - 1:                                |
|                                                                       |
| 4\. Compute partner ⟵ i XOR j.                                        |
|                                                                       |
| 5\. If partner \<= i, continue to the next index.                     |
|                                                                       |
| 6\. If (i AND k) = 0 and A\[i\] \> A\[partner\]:                      |
|                                                                       |
| 7\. swap A\[i\] and A\[partner\].                                     |
|                                                                       |
| 8\. If (i AND k) != 0 and A\[i\] \< A\[partner\]:                     |
|                                                                       |
| 9\. swap A\[i\] and A\[partner\].                                     |
|                                                                       |
| 10\. Return A.                                                        |
+-----------------------------------------------------------------------+

У цьому псевдокоді цикл за k визначає розмір блока, а цикл за j - окремі
етапи порівняння в межах цього блока. Для кожного індексу i обчислюється
парний індекс partner, після чого виконується compare-swap у потрібному
напрямку. Такий запис зручний тим, що майже безпосередньо переходить у
програмну реалізацію і наочно показує незалежність порівнянь усередині
одного шару мережі. Наведений псевдокод узагальнює ітеративне подання
алгоритму, описане в \[2-4\].

## 1.3 Відомі паралельні реалізації алгоритму

Історично бітонічне сортування розглядалося як алгоритм, зручний для
паралельного виконання на обчислювальних структурах із регулярною
топологією \[1, 2, 5\]. Регулярність структури компараторів дає змогу
реалізовувати його як на SIMD-подібних моделях, так і на
багатопроцесорних або багатопотокових системах.

Найбільш загальна ідея паралельної реалізації полягає в тому, що всі
compare-swap операції одного шару мережі виконуються одночасно. Після
завершення шару виконується синхронізація, і лише тоді починається
наступний етап \[3, 4\].

У літературі можна виділити кілька типових підходів до паралельної
реалізації bitonic sort. Перший полягає у прямому відображенні
компараторів мережі на паралельні обчислювальні елементи, де кожен потік
або процес виконує свою підмножину compare-swap операцій \[2, 3, 5\].
Другий підхід характерний для систем зі спільною пам\'яттю, де
виконавцям передаються діапазони індексів, а синхронізація здійснюється
між окремими шарами мережі. Третій підхід орієнтований на GPU та
архітектури масового паралелізму, де один шар мережі виконується великою
кількістю потоків одночасно \[3, 4, 6\].

Окремий інтерес становлять CUDA-реалізації алгоритму. У відповідній
публікації розглядається підхід, де продуктивність підвищується за
рахунок використання shared memory та зменшення кількості kernel
launches \[6\]. Це джерело корисне як приклад практичного напряму
оптимізації паралельних реалізацій, хоча його результати не варто
безпосередньо порівнювати з реалізацією на Python.

Схематично один із підходів до розподілу компараторів між виконавцями на
окремому паралельному етапі показано на рис. 1.2. Ця схема підкреслює,
що незалежні compare-swap операції одного шару можуть бути закріплені за
різними процесами або потоками без зміни логіки сортувальної мережі \[2,
4\].

# 2 РОЗРОБКА ПОСЛІДОВНОГО АЛГОРИТМУ ТА АНАЛІЗ ЙОГО ШВИДКОДІЇ

## 2.1 Проєктування послідовної реалізації

Для реалізації було обрано ітеративну форму алгоритму бітонічного
сортування. Такий вибір узгоджується з псевдокодом, наведеним у
попередньому розділі, і дає змогу безпосередньо відобразити шари
сортувальної мережі у вкладені цикли програми. Зовнішній цикл відповідає
змінюванню розміру блока k, а внутрішній - змінюванню кроку j, що
визначає пари елементів для compare-swap операцій.

Оскільки класичний алгоритм bitonic sort природно визначений для масивів
розміру 2\^k, у реалізації передбачено попередню підготовку вхідних
даних. Якщо кількість елементів не є степенем двійки, масив доповнюється
до найближчої більшої такої довжини службовими значеннями, більшими за
максимум вхідного масиву. Після завершення сортування ці додані елементи
відкидаються. Такий підхід дає змогу зберегти правильність роботи
алгоритму та водночас підтримати обробку вхідних даних довільної
довжини.

Для коректного порівняння з майбутньою паралельною реалізацією
послідовний алгоритм виконує ті самі базові compare-swap операції над
тими самими індексами, що й паралельний. Різниця між реалізаціями надалі
полягатиме лише в способі розподілу цих операцій між виконавцями, а не в
математичній суті алгоритму. Саме тому послідовна реалізація виступає не
тільки як самостійний програмний модуль, а і як еталон для перевірки
коректності та вимірювання прискорення.

## 2.2 Програмна реалізація послідовного алгоритму

Послідовна реалізація розміщена в модулі SequentialBitonicSorter.
Основна логіка сортування винесена в окрему функцію
\_bitonic_sort_iterative, яка виконує сортування in-place над робочим
масивом. На рівні класу реалізовано підготовку вхідних даних, запуск
сортування, обрізання доповнення та, за потреби, формування результату у
спадному порядку.

Базовий клас BitonicSorter містить допоміжні методи для обчислення
найближчого степеня двійки, доповнення масиву та вимірювання часу
виконання. Такий поділ спрощує структуру коду і дає змогу
використовувати однакові підготовчі дії як для послідовної, так і для
паралельної реалізацій.

У послідовній версії використано array.array(\'q\'), тобто компактне
зберігання 64-бітних цілих чисел. Це рішення зменшує накладні витрати
пам'яті порівняно зі звичайним списком Python і покращує локальність
доступу до даних. Додатково в середині внутрішнього циклу
використовується локальне посилання на масив, щоб зменшити кількість
зайвих звернень до атрибутів і трохи скоротити накладні витрати
інтерпретатора.

Логіка compare-swap операції реалізована так: для кожного індексу i
обчислюється партнер partner = i XOR j. Якщо partner \<= i, пара
пропускається, щоб не виконувати одне й те саме порівняння двічі. Далі
за допомогою виразу (i & k) == 0 визначається, чи повинна відповідна
підпослідовність на поточному етапі впорядковуватися за зростанням чи за
спаданням. Якщо порядок порушено, елементи міняються місцями.

Такий підхід добре відображає структуру сортувальної мережі: кожний шар
мережі відповідає фіксованому значенню j, а повний етап побудови або
злиття бітонічної послідовності - фіксованому значенню k. Отже,
програмний код безпосередньо реалізує ту ж саму схему порівнянь, яка
була описана в теоретичній частині[. Повний код надано у Додатку
БУКВА]{.mark}

j \>\>= 1

## 2.3 Тестування та доведення коректності

Для перевірки коректності було підготовлено набір тестів, що охоплює як
прості, так і нетривіальні вхідні дані. Послідовна реалізація

## 3.1 Характеристика обраної технології

Python є мовою високого рівня з розвиненою стандартною бібліотекою, у
якій уже наявні засоби для побудови багатопотокових і багатопроцесних
програм \[7, 9, 10\]. Для цієї курсової роботи це важливо з двох причин.
По-перше, реалізація алгоритму бітонічного сортування потребує
багаторазового відтворення однакових compare-swap операцій, тому код
повинен бути компактним і достатньо прозорим для аналізу. По-друге,
експериментальна частина роботи вимагає не лише реалізації алгоритму, а
й тестування, вимірювання часу виконання та варіювання параметрів
паралельного запуску.

Попри те, що Python поступається мовам нижчого рівня за максимальною
швидкодією, він добре підходить для прототипування та дослідження
паралельних алгоритмів. У межах цієї роботи це дозволяє зосередитися на
правильному поділі задачі, синхронізації процесів і спільному доступі до
даних, не ускладнюючи реалізацію низькорівневими деталями керування
пам\'яттю \[7, 8\].

## 3.2 Засоби для організації паралельних обчислень

Основним засобом паралельної реалізації в роботі обрано модуль
multiprocessing. Його головна перевага полягає в тому, що він
використовує окремі процеси, а не потоки, і тому дає змогу задіювати
кілька ядер процесора для CPU-bound задач \[7\]. Для задачі бітонічного
сортування це принципово, оскільки сортування є саме таким типом
обчислень.

Важливим доповненням до multiprocessing є модуль
multiprocessing.shared_memory, який надає засоби спільного доступу до
даних між процесами. Для паралельних обчислень це корисно тоді, коли
потрібно працювати з великими масивами даних і водночас зменшити
накладні витрати на їх передавання між процесами. Документація цього
модуля прямо підкреслює, що спільна пам\'ять дає змогу уникати
серіалізації, десеріалізації та зайвого копіювання даних \[8\].

Поряд з multiprocessing у Python існує модуль threading, який забезпечує
запуск кількох потоків у межах одного процесу \[9\]. Його перевагою є
спільний адресний простір і простіша модель обміну даними. Проте для
CPU-bound задач у стандартній реалізації CPython використання потоків
має суттєве обмеження через GIL: одночасне виконання Python-байткоду на
кількох ядрах фактично не досягається. Тому threading доцільніше
застосовувати для I/O-bound задач, а не для інтенсивного сортування
великих масивів \[9\].

Як додаткову альтернативу варто розглянути модуль concurrent.futures.
Він надає високорівневий інтерфейс ThreadPoolExecutor і
ProcessPoolExecutor для запуску задач у потоках або процесах \[10\].
Перевага цього підходу полягає в простішому програмному інтерфейсі й
уніфікованій роботі з пулом виконавців. Проте в межах цієї роботи
важливо не лише запускати паралельні задачі, а й мати достатньо явний
контроль над процесною моделлю, синхронізацією та організацією спільного
доступу до даних. Тому concurrent.futures доцільно розглядати як корисну
альтернативу, але не як основний інструмент роботи.

## 3.3 Обґрунтування вибору для цієї задачі

Алгоритм бітонічного сортування має регулярну структуру, у якій значна
частина операцій може виконуватися незалежно. Разом з тим між окремими
етапами обчислень потрібне узгодження, щоб зберігалася правильна
послідовність перетворень даних. Отже, для цієї задачі потрібен
інструмент, який дозволяє використовувати кілька ядер процесора, ділити
роботу на підзадачі та підтримувати контрольований обмін даними між
виконавцями.

Саме цим вимогам найкраще відповідає зв\'язка multiprocessing +
multiprocessing.shared_memory. Вона поєднує справжній процесний
паралелізм із можливістю організувати спільний доступ до даних без
переходу до зовнішніх платформ або спеціалізованих фреймворків \[7, 8\].
Для навчальної курсової роботи це є важливою перевагою, оскільки обрані
засоби входять до стандартної бібліотеки Python, добре документовані та
достатні для побудови й дослідження паралельного алгоритму на
багатоядерному CPU.

Отже, для даної курсової роботи вибір Python як мови реалізації,
multiprocessing як основного механізму паралелізації та
multiprocessing.shared_memory як засобу спільного доступу до даних є
обґрунтованим \[7, 8\]. Модулі threading і concurrent.futures
розглядаються як корисні альтернативи, однак для CPU-bound сортування
великих масивів вони або поступаються за практичною придатністю, або
менш точно відповідають вимогам задачі \[9, 10\].

## 4.1 Проєктування паралельного алгоритму

Проєктування паралельного алгоритму доцільно починати не з коду, а з
аналізу сортувальної мережі bitonic sort. Якщо розглянути схему на рис.
4.1, то можна побачити, що мережа складається з послідовності колонок
компараторів, кожна з яких відповідає окремому кроку для фіксованих
значень k і j. Усередині однієї колонки всі compare-swap операції є
незалежними, оскільки працюють з різними парами елементів. Отже, саме
колонка компараторів є природною паралельною фазою, а перехід від однієї
колонки до наступної - природною межею синхронізації. Такий погляд
узгоджується з описом сортувальних мереж і паралельної формалізації
bitonic sort у \[11\].

Після виявлення цих паралельних фаз потрібно визначити, яким саме
способом організувати декомпозицію задачі. Теоретично можливі кілька
варіантів. Перший варіант - трактувати кожен окремий компаратор як
самостійну паралельну задачу. Такий підхід максимально розкриває наявний
паралелізм мережі, але на практиці створює надто дрібні задачі, для яких
накладні витрати на керування можуть бути більшими за корисну роботу.
Другий варіант - розпаралелювати великі структурні частини алгоритму,
наприклад цілі фрагменти побудови або злиття бітонічних послідовностей.
Однак у цьому разі ускладнюється узгодження між підзадачами, а сама
схема стає менш регулярною.

Найбільш збалансованим виявляється третій підхід: розглядати одну
колонку мережі як окрему паралельну фазу, а множину індексів усередині
цієї фази ділити між виконавцями за неперекривними діапазонами. У такій
постановці зберігається вихідна структура алгоритму, не втрачається
наочний зв\'язок із сортувальною мережею, а навантаження між виконавцями
можна розподіляти передбачувано і без дублювання compare-swap операцій.

Корисним для проєктування є також узагальнення, наведене у \[11\]:
bitonic sort можна розглядати або у формулюванні one element per
process, або у формулюванні a block of elements per process. Перший
підхід добре підходить для теоретичного аналізу мережі, оскільки кожна
лінія мережі інтерпретується як окремий процес. Проте для практичної
реалізації на звичайному багатоядерному CPU цей варіант не є зручним, бо
кількість елементів масиву значно перевищує кількість доступних
процесів. Тому для прикладної реалізації природніше переходити до
блокового підходу, у якому кожен виконавець обробляє не один елемент, а
частину спільного масиву.

Окремим питанням проєктування є синхронізація. Оскільки результат кожної
колонки компараторів стає входом для наступної, між паралельними фазами
необхідно вводити точку узгодження. Це означає, що паралельний bitonic
sort не є сукупністю повністю незалежних задач, а являє собою
послідовність паралельних етапів, розділених бар\'єрами синхронізації.
Саме тому місця потенційної паралелізації доцільно позначати покроково:
колонка - це зона паралельного виконання, межа між колонками - момент
синхронізації.

Ще одним важливим аспектом є модель зберігання даних. Якщо кожен
виконавець працює з власною копією масиву, то виникають зайві витрати на
передавання й подальше злиття даних. Тому на етапі проєктування доцільно
орієнтуватися на спільний буфер, у якому різні виконавці обробляють
різні частини одного масиву. Такий підхід краще відповідає і регулярній
структурі сортувальної мережі, і вимогам до практичної реалізації на
багатоядерному процесорі \[8\].

З позиції часової складності ключовим є те, що bitonic sorting network
для n = 2\^m має m стадій, а i-та стадія містить i колонок компараторів.
Тому загальна кількість послідовних кроків мережі дорівнює

$$d(n) = 1 + 2 + \ldots + \log_{2}n = \frac{\left( \log_{2}n\left( \log_{2}n + 1 \right) \right)}{2} = ϴ\left( \log^{2}n \right)$$

Саме цю оцінку для глибини bitonic network і кількості compare-exchange
кроків наводять Грама та співавтори \[11\]. У моделі one element per
process, коли кожен процес відповідає одній лінії мережі, паралельний
час в ідеалізованій постановці оцінюється як

$$T_{p}(n) = ϴ\left( \log^{2}n \right),\ p = n$$

Для практичної реалізації, у якій p \<\< n, більш доречною є блокова
інтерпретація. У ній на кожному кроці сортувальної мережі всі n / 2
compare-swap операцій розподіляються між p процесами, тобто кожен процес
у середньому обробляє $ϴ$(n / p) елементів або $ϴ$(n / p) локальної
роботи за один крок. Оскільки таких кроків $ϴ$(log\^2 n), отримуємо
оцінку

$$T_{p}(n) = ϴ(\frac{n}{p}\ \log^{2}n)$$

У реальній програмній реалізації до цієї оцінки додаються витрати на
синхронізацію між сусідніми колонками мережі. Якщо позначити вартість
однієї точки узгодження як $T_{sync}$, то більш наближено час виконання
можна подати у вигляді

$$T_{p}^{*}(n) = ϴ(\frac{n}{p}\log^{2}{n + T_{sync}\log^{2}{n)}}$$

Отже, теоретичний виграш від паралелізації визначається тим, що основна
частина compare-swap операцій усередині кожної колонки може виконуватися
одночасно, тоді як обмеженням залишаються лише послідовні переходи між
колонками та накладні витрати на синхронізацію. Саме тому bitonic sort
добре підходить для паралельної реалізації, але водночас не демонструє
ідеального лінійного масштабування для малих обсягів даних.

Таким чином, у процесі проєктування паралельного bitonic sort було
сформульовано такі базові рішення: використовувати сортувальну мережу як
засіб виявлення паралельних фаз; трактувати кожну колонку компараторів
як окремий крок паралельної обробки; ділити індекси між виконавцями за
неперекривними діапазонами; орієнтуватися на блокову обробку даних, а не
на модель один елемент -- один процес; вводити синхронізацію між
сусідніми колонками мережі; використовувати спільний масив даних замість
незалежних локальних копій. Саме ці проектні міркування стали основою
подальшої реалізації. На рисунку 4.3 зображено візуалізацію обраного

## 4.2 Реалізація паралельного алгоритму

Паралельна реалізація розміщена в окремому модулі та оформлена як клас
ParallelBitonicSorter, сумісний з базовим інтерфейсом сортувальника.
Такий підхід дозволяє використовувати однаковий спосіб виклику для
послідовної та паралельної версій, що спрощує тестування і подальше
експериментальне порівняння.

Для організації паралельних обчислень використано модуль
multiprocessing, а для спільного доступу до масиву даних -
multiprocessing.shared_memory \[7, 8\]. Під час запуску формується пул
робочих процесів, які під\'єднуються до спільного буфера пам\'яті та
надалі багаторазово використовуються при проходженні послідовності шарів
bitonic sort. Це дозволяє уникнути зайвого створення процесів на кожному
кроці алгоритму.

Розподіл роботи між процесами виконується статично: повний діапазон
індексів ділиться на кілька частин, після чого для кожного кроку
алгоритму формується набір задач однакової структури. Кожна задача
містить межі фрагмента масиву та параметри поточного шару сортування.
Після завершення всіх задач для поточного шару програма переходить до
наступного шару. Така схема реалізації добре узгоджується з регулярною
структурою bitonic sort і не потребує складного динамічного
балансування.

Важливо, що паралельна реалізація зберігає сумісність із послідовною
версією також на рівні підготовки даних. Якщо довжина вхідного масиву не
є степенем двійки, він так само доповнюється до найближчої відповідної
довжини. Після завершення сортування службові елементи відкидаються.
Завдяки цьому результати обох реалізацій залишаються безпосередньо
порівнюваними. Розподіл compare-swap операцій між 4 процесами для
bitonic sort на 16 елементах показано на рис. 4.4.

arr: ctypes.Array\[ctypes.c_int64\],

start: int,

end: int,

) -\> None:

arr_local = arr

for i in range(start, end):

partner = i \^ stride

if partner \<= i:

continue

vi = arr_local\[i\]

vp = arr_local\[partner\]

if (vi \> vp) == ((i & size) == 0):

arr_local\[i\], arr_local\[partner\] = vp, vi

## 4.3 Тестування паралельного алгоритму

Перевірка коректності паралельної реалізації виконується за двома
напрямами. Перший напрям охоплює звичайні функціональні тести: порожній
масив, один елемент, вже відсортовані та обернено відсортовані
послідовності, дублікати, від'ємні числа, не степеневу довжину масиву,
сортування за спаданням та незмінність вхідного списку. Другий напрям є
специфічним саме для паралельної реалізації й перевіряє коректність
роботи за різної кількості процесів, а також збіг результатів із
послідовною реалізацією.

Окремо перевіряється ситуація, коли кількість процесів перевищує
кількість елементів масиву, а також випадки з великими випадковими
масивами. Для доведення еквівалентності реалізацій використовуються
спеціальні тести, у яких результат паралельного алгоритму безпосередньо
порівнюється з результатом послідовного сортування на тих самих вхідних
даних. Такий підхід є особливо важливим, оскільки саме послідовна

## 5.1 Методика експериментального дослідження

Експерименти проводилися на платформі macOS-15.7.3-arm64-arm-64bit із
процесором Apple M1 Pro з 8 ядрами продуктивності CPU та інтерпретатором
Python 3.12.12. Для дослідження використовувалися масиви розмірів від 16
до 4 194 304 елементів. Паралельна реалізація тестувалася для 2, 4 та 8
процесів.

Для кожної конфігурації виконувалося 12 запусків, з яких 2 були
прогрівальними, а 10 використовувалися для статистичного аналізу. Після
цього для усунення аномальних значень застосовувалося IQR-фільтрування
(1.5x IQR), а основним показником часу виконання обиралася медіана.
Додатково обчислювався напівінтервал довіри, що дозволяло оцінити
стабільність отриманих результатів.

Тестові масиви генерувалися автоматично з фіксованим зерном випадковості
42, причому значення елементів належали діапазону \[-100000; 100000\].
Генерація вхідних даних виконувалася до початку вимірювання часу, тому в
результатах враховується саме час сортування. Для паралельної реалізації
до цього часу входять усі супровідні витрати алгоритму: доповнення до
степеня двійки, створення та використання спільної пам\'яті, розподіл
задач між процесами та синхронізація між шарами сортувальної мережі.

Для оцінки результатів використовувалися стандартні показники
паралельних обчислень \[11\]:

$$S_{p}\  = \frac{T_{1}}{T_{p}},\ $$

де $T_{1}$ -- час послідовного виконання, а $T_{p}$ -- час виконання на
*p* процесах.

Ефективність визначається за формулою

$$E_{p}\  = \frac{S_{p}}{p}\ $$

Таким чином, розглянуті експерименти дозволяють оцінити не лише сам факт
прискорення, а й ступінь корисного використання додаткових
обчислювальних ресурсів.

## 5.2 Залежність часу виконання від розміру вхідних даних

Отримані результати підтверджують очікувану закономірність: зі
збільшенням розміру вхідного масиву час виконання як послідовної, так і
паралельної реалізацій зростає. Для малих масивів паралельний варіант
працює повільніше через накладні витрати на створення процесів,
організацію спільної пам\'яті та синхронізацію. Наприклад, для 1024
елементів послідовна реалізація виконується приблизно за 0,0040 с, тоді
як паралельна навіть для 2 процесів потребує близько 0,0511 с.

## 5.3 Аналіз прискорення та ефективності

Розрахунок прискорення показує, що для малих вхідних масивів воно є
меншим за одиницю, тобто паралельна реалізація в таких умовах не дає
практичного виграшу. Це природний результат, оскільки накладні витрати
на запуск процесів і синхронізацію виявляються більшими, ніж час власне
сортування.

Починаючи приблизно з 16 384 елементів, прискорення перевищує 1, а зі
збільшенням розміру масиву далі зростає. Для 4 процесів прискорення
змінюється від 1,229 на 16 384 елементах до 3,582 на 4 194 304
елементах. Для 8 процесів воно становить 1,034 на 16 384 елементах,
3,774 на 524 288 елементах і досягає максимального значення 4,642 на 4
194 304 елементах. Для 2 процесів максимальне зафіксоване прискорення
становить 1,870.

Аналіз ефективності показує, що з точки зору відносного використання
ресурсів найкращий результат дає запуск на 2 процесах: для найбільших
масивів ефективність сягає 0,935. Для 4 процесів максимальна
ефективність становить 0,896, а для 8 процесів - 0,580. Це означає, що
зі збільшенням кількості процесів абсолютний виграш у швидкодії зростає,
однак частка корисно використаних ресурсів зменшується через додаткові
витрати на координацію процесів.

Таким чином, результати експериментів демонструють типовий компроміс
паралельних обчислень: збільшення кількості процесів дозволяє отримати
більше прискорення на великих обсягах даних, але водночас знижує
ефективність використання кожного окремого процесу. Залежність
прискорення від розміру вхідного масиву показано на рис. 5.2.

# ВИСНОВКИ

У ході виконання курсової роботи було досліджено алгоритм бітонічного
сортування та особливості його паралельної реалізації на багатоядерних
обчислювальних системах. У теоретичній частині розглянуто принцип
побудови бітонічних послідовностей, структуру сортувальної мережі та
причини, через які bitonic sort є зручним для розпаралелювання. На
основі проведеного аналізу було показано, що регулярність compare-swap
операцій і фіксована послідовність шарів мережі створюють природну
основу для побудови паралельного алгоритму.

У практичній частині роботи розроблено послідовну та паралельну
реалізації алгоритму мовою Python. Послідовна версія була використана як
базова для перевірки коректності та подальшого порівняння швидкодії.
Паралельна реалізація побудована з використанням процесів і спільної
пам\'яті, що дало змогу виконувати незалежні compare-swap операції
всередині окремих фаз алгоритму одночасно та зберегти логіку вихідного
методу сортування. Проведене тестування підтвердило коректність обох
реалізацій на граничних, типових і великих наборах даних.

Експериментальне дослідження показало, що паралельна реалізація не є
доцільною для малих масивів через накладні витрати на створення
процесів, синхронізацію та обмін даними. Водночас починаючи приблизно з
16 384 елементів вона починає демонструвати стійкий практичний виграш у
швидкодії порівняно з послідовною версією. Найбільше прискорення в
проведених експериментах було досягнуто для конфігурації з 8 процесами
на масиві розміру 4 194 304 елементів і становило 4,642, а найвища
ефективність спостерігалася для 2 процесів і досягала 0,935. Таким
чином, встановлено, що мету роботи досягнуто: паралельну реалізацію
алгоритму бітонічного сортування розроблено, протестовано й
експериментально підтверджено її ефективність, причому отримане
прискорення перевищує цільове значення 1,2 для достатньо великих обсягів
даних.

Отримані результати можуть бути використані як основа для подальших
досліджень паралельних методів сортування, зокрема для порівняння з
іншими алгоритмами, для вивчення впливу архітектури обчислювальної
системи на масштабованість, а також для подальшої оптимізації реалізації
на багатоядерних CPU або інших платформах паралельних обчислень.

def compute_stats(times: list\[float\]) -\> RunStats:

n = len(times)

ci_half = (1.96 \* stdev / math.sqrt(n)) if n \>= 2 else 0.0

par_stats.median)
