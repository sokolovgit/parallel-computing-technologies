## Робочий текст розділу

Метою цього розділу є експериментально підтвердити, що паралельна реалізація алгоритму бітонічного сортування забезпечує практичний виграш у швидкодії порівняно з послідовною версією. Для цього необхідно не лише порівняти абсолютний час виконання, а й проаналізувати, як змінюються прискорення та ефективність залежно від розміру вхідного масиву і кількості процесів.

### 5.1 Методика експериментального дослідження

Експерименти проводилися на платформі `macOS-15.7.3-arm64-arm-64bit` з `8` ядрами CPU та інтерпретатором `Python 3.12.12`. Для дослідження використовувалися масиви розмірів від `16` до `4 194 304` елементів. Паралельна реалізація тестувалася для `2`, `4` та `8` процесів.

Для кожної конфігурації виконувалося `12` запусків, з яких `2` були прогрівальними, а `10` використовувалися для статистичного аналізу. Після цього для усунення аномальних значень застосовувалося IQR-фільтрування (`1.5x IQR`), а основним показником часу виконання обиралася медіана. Додатково обчислювався напівінтервал довіри, що дозволяло оцінити стабільність отриманих результатів.

Тестові масиви генерувалися автоматично з фіксованим зерном випадковості `42`, причому значення елементів належали діапазону `[-100000; 100000]`. Генерація вхідних даних виконувалася до початку вимірювання часу, тому в результатах враховується саме час сортування. Для паралельної реалізації до цього часу входять усі супровідні витрати алгоритму: доповнення до степеня двійки, створення та використання спільної пам’яті, розподіл задач між процесами та синхронізація між шарами сортувальної мережі.

Для оцінки результатів використовувалися стандартні показники паралельних обчислень:

`S_p = T_1 / T_p`,

де `T_1` - час послідовного виконання, а `T_p` - час виконання на `p` процесах.

Ефективність визначалася за формулою

`E_p = S_p / p`.

Таким чином, розглянуті експерименти дозволяють оцінити не лише сам факт прискорення, а й ступінь корисного використання додаткових обчислювальних ресурсів.

### 5.2 Залежність часу виконання від розміру вхідних даних

Отримані результати підтверджують очікувану закономірність: зі збільшенням розміру вхідного масиву час виконання як послідовної, так і паралельної реалізацій зростає. Для малих масивів паралельний варіант працює повільніше через накладні витрати на створення процесів, організацію спільної пам’яті та синхронізацію. Наприклад, для `1024` елементів послідовна реалізація виконується приблизно за `0,0040` с, тоді як паралельна навіть для `2` процесів потребує близько `0,0511` с.

Зі збільшенням обсягу даних ситуація змінюється. Для масиву з `16 384` елементів уже спостерігається перший практичний виграш: для `4` процесів час виконання становить `0,1038` с проти `0,1276` с у послідовному випадку. Для `131 072` елементів послідовна реалізація виконується за `1,4737` с, тоді як паралельна - за `0,8602` с (`2` процеси), `0,5115` с (`4` процеси) та `0,4858` с (`8` процесів). На найбільшому дослідженому масиві розміру `4 194 304` елементів час послідовного виконання становить `77,8639` с, а паралельного - `41,6354` с, `21,7374` с та `16,7741` с відповідно для `2`, `4` і `8` процесів.

Отже, паралельна реалізація стає доцільною не для всіх розмірів масивів, а лише тоді, коли корисна обчислювальна робота починає переважати витрати на керування паралельними процесами. У межах проведених експериментів така межа починає проявлятися приблизно від `16 384` елементів і далі стає дедалі виразнішою.

### 5.3 Аналіз прискорення та ефективності

Розрахунок прискорення показує, що для малих вхідних масивів воно є меншим за одиницю, тобто паралельна реалізація в таких умовах не дає практичного виграшу. Це природний результат, оскільки накладні витрати на запуск процесів і синхронізацію виявляються більшими, ніж час власне сортування.

Починаючи приблизно з `16 384` елементів, прискорення перевищує `1`, а зі збільшенням розміру масиву далі зростає. Для `4` процесів прискорення змінюється від `1,229` на `16 384` елементах до `3,582` на `4 194 304` елементах. Для `8` процесів воно становить `1,034` на `16 384` елементах, `3,774` на `524 288` елементах і досягає максимального значення `4,642` на `4 194 304` елементах. Для `2` процесів максимальне зафіксоване прискорення становить `1,870`.

Аналіз ефективності показує, що з точки зору відносного використання ресурсів найкращий результат дає запуск на `2` процесах: для найбільших масивів ефективність сягає `0,935`. Для `4` процесів максимальна ефективність становить `0,896`, а для `8` процесів - `0,580`. Це означає, що зі збільшенням кількості процесів абсолютний виграш у швидкодії зростає, однак частка корисно використаних ресурсів зменшується через додаткові витрати на координацію процесів.

Таким чином, результати експериментів демонструють типовий компроміс паралельних обчислень: збільшення кількості процесів дозволяє отримати більше прискорення на великих обсягах даних, але водночас знижує ефективність використання кожного окремого процесу.

### 5.4 Вплив кількості процесів на результат

Порівняння конфігурацій із `2`, `4` та `8` процесами показує, що оптимальна кількість процесів залежить від розміру вхідних даних. Для малих і середніх масивів конфігурація з `4` процесами часто виявляється кращою за `8`, оскільки має менші накладні витрати. Наприклад, для `32 768` елементів час виконання становить `0,2162` с для `2` процесів, `0,1532` с для `4` процесів і `0,1712` с для `8` процесів. Аналогічно, для `65 536` елементів конфігурація з `4` процесами (`0,2608` с) ще залишається трохи кращою за конфігурацію з `8` процесами (`0,2690` с).

Починаючи з більших розмірів масиву, перевага переходить до `8` процесів. Уже на `131 072` елементах найкращий час показує саме конфігурація з `8` процесами (`0,4858` с), а для всіх більших масивів вона залишається найшвидшою. Це означає, що зі зростанням обсягу даних витрати на координацію процесів стають відносно меншими, і система може ефективніше використовувати більшу кількість ядер.

З практичного погляду можна зробити такі висновки. Якщо потрібно мінімізувати накладні витрати й отримати високу ефективність, доцільно використовувати `2` або `4` процеси. Якщо ж ціллю є максимальне прискорення на великих масивах, найкращі результати забезпечує конфігурація з `8` процесами. Саме вона дає максимальний виграш у швидкодії для дослідженого обладнання.

### 5.5 Висновки до розділу

Експериментальне дослідження показало, що паралельна реалізація алгоритму бітонічного сортування дійсно забезпечує відчутний практичний виграш, але лише для достатньо великих масивів. Для малих вхідних даних переважають накладні витрати на організацію паралельних обчислень, тому послідовний варіант є швидшим. Починаючи приблизно з `16 384` елементів, паралельна реалізація починає демонструвати стабільну перевагу.

Найбільше прискорення в проведених експериментах було досягнуто для конфігурації з `8` процесами на масиві розміру `4 194 304` елементів і становило `4,642`. Найвища ефективність спостерігалася для `2` процесів і досягала `0,935`, що свідчить про менші накладні витрати при помірному рівні паралелізму. Отже, залежно від мети можна рекомендувати різні конфігурації запуску: `2-4` процеси для більш ефективного використання ресурсів і `8` процесів для отримання максимального прискорення на великих наборах даних.
