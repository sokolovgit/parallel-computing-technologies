# 1 ОПИС АЛГОРИТМУ ТА ЙОГО ВІДОМИХ ПАРАЛЕЛЬНИХ РЕАЛІЗАЦІЙ

Бітонічне сортування належить до тих алгоритмів, у яких ідея побудови послідовності порівнянь тісно пов'язана з можливістю ефективного паралельного виконання. На відміну від багатьох інших методів сортування, тут порядок compare-swap операцій визначається наперед і може бути поданий у вигляді сортувальної мережі. Саме тому для коректного переходу до програмної реалізації важливо спочатку розглянути теоретичну основу алгоритму, спосіб формування бітонічних послідовностей, подання у вигляді псевдокоду та типові підходи до його паралельної реалізації.

## 1.1 Загальна характеристика алгоритму бітонічного сортування

Бітонічне сортування належить до алгоритмів сортування порівняннями та є класичним прикладом сортувальної мережі (рис. 1.1). Алгоритм був запропонований Кеннетом Бетчером [2]. Його ідея полягає в тому, що довільну послідовність можна впорядковувати через побудову бітонічних послідовностей і їх подальше бітонічне злиття.

Бітонічною називають послідовність, яка спочатку монотонно зростає, а потім монотонно спадає, або може бути отримана циклічним зсувом із такої послідовності [3, 4]. Для двох елементів така послідовність утворюється тривіально, а далі з малих бітонічних підпослідовностей послідовно формуються більші.

Особливість алгоритму полягає в тому, що послідовність компараторів задається наперед і залежить лише від розміру масиву. У термінах Кнута це відповідає oblivious sequence of comparisons, де структура порівнянь є фіксованою, а окремі компараторні модулі можна подати як вузли мережі [2, 5].

У класичному варіанті алгоритм працює для кількості елементів, що дорівнює степеню двійки. Для масивів іншої довжини на практиці застосовують доповнення до найближчого значення вигляду `2^k`.

Класичне подання bitonic sort часто є рекурсивним: дві половини масиву впорядковуються у протилежних напрямках, утворюючи бітонічну послідовність, після чого виконується бітонічне злиття [2, 3, 4]. Для практичної реалізації зручнішою є ітеративна форма, оскільки вона подає алгоритм як послідовність шарів compare-swap операцій. У такому записі використовуються параметри `k` і `j`, а партнер для індексу `i` визначається як `i XOR j`. Саме цей варіант краще підходить для програмної реалізації, побудови мережі та розподілу роботи між паралельними виконавцями.

Асимптотична складність бітонічного сортування в послідовному виконанні становить `O(n log^2 n)` [3, 4]. Воно поступається оптимальним послідовним алгоритмам сортування, але компенсує це регулярною структурою, передбачуваним шаблоном доступу до даних і зручністю розпаралелювання. На кожному етапі значна частина compare-swap операцій є незалежною, тому вони можуть виконуватися одночасно, а синхронізація потрібна лише між окремими шарами мережі [1, 3, 4].

## 1.2 Псевдокод алгоритму

Для опису алгоритму в цій роботі доцільно використати ітеративний псевдокод, який безпосередньо відповідає структурі сортувальної мережі та зручний для подальшого переходу до паралельної реалізації. Такий запис узгоджується з вимогами методичних рекомендацій: у першому рядку вказується назва алгоритму, далі подаються вхідні та вихідні дані, а самі дії описуються англійською мовою, близькою до природної.

```text
Algorithm Iterative bitonic sorting
Input: A - array of elements, n - number of elements, where n is a power of 2.
Output: array A sorted in ascending order.

1. For each block size k = 2, 4, ..., n:
2.     For each stride j = k / 2, k / 4, ..., 1:
3.         For each index i = 0, ..., n - 1:
4.             Compute partner <- i XOR j.
5.             If partner <= i, continue to the next index.
6.             If (i AND k) = 0 and A[i] > A[partner]:
7.                 swap A[i] and A[partner].
8.             If (i AND k) != 0 and A[i] < A[partner]:
9.                 swap A[i] and A[partner].
10. Return A.
```

У цьому псевдокоді цикл за `k` визначає розмір блока, а цикл за `j` - окремі етапи порівняння в межах цього блока. Для кожного індексу `i` обчислюється парний індекс `partner`, після чого виконується compare-swap у потрібному напрямку. Такий запис зручний тим, що майже безпосередньо переходить у програмну реалізацію і наочно показує незалежність порівнянь усередині одного шару мережі. Наведений псевдокод узагальнює ітеративне подання алгоритму, описане в [2-4].

## 1.3 Відомі паралельні реалізації алгоритму

Історично бітонічне сортування розглядалося як алгоритм, зручний для паралельного виконання на обчислювальних структурах із регулярною топологією [1, 2, 5]. Регулярність структури компараторів дає змогу реалізовувати його як на SIMD-подібних моделях, так і на багатопроцесорних або багатопотокових системах.

Найбільш загальна ідея паралельної реалізації полягає в тому, що всі compare-swap операції одного шару мережі виконуються одночасно. Після завершення шару виконується синхронізація, і лише тоді починається наступний етап [3, 4].

У літературі можна виділити кілька типових підходів до паралельної реалізації bitonic sort. Перший полягає у прямому відображенні компараторів мережі на паралельні обчислювальні елементи, де кожен потік або процес виконує свою підмножину compare-swap операцій [2, 3, 5]. Другий підхід характерний для систем зі спільною пам'яттю, де виконавцям передаються діапазони індексів, а синхронізація здійснюється між окремими шарами мережі. Третій підхід орієнтований на GPU та архітектури масового паралелізму, де один шар мережі виконується великою кількістю потоків одночасно [3, 4, 6].

Окремий інтерес становлять CUDA-реалізації алгоритму. У відповідній публікації розглядається підхід, де продуктивність підвищується за рахунок використання shared memory та зменшення кількості kernel launches [6]. Це джерело корисне як приклад практичного напряму оптимізації паралельних реалізацій, хоча його результати не варто безпосередньо порівнювати з реалізацією на Python.

Схематично один із підходів до розподілу компараторів між виконавцями на окремому паралельному етапі показано на рис. 1.2. Ця схема підкреслює, що незалежні compare-swap операції одного шару можуть бути закріплені за різними процесами або потоками без зміни логіки сортувальної мережі [2, 4].

## 1.4 Висновки до розділу

У першому розділі розглянуто теоретичну основу алгоритму бітонічного сортування, його місце серед сортувальних мереж і причини, через які він добре підходить для паралельної реалізації. На основі опрацьованих джерел показано, що регулярна структура compare-swap операцій і фіксована послідовність шарів мережі дають змогу природно перейти від теоретичного опису до програмної реалізації [1-5].

Огляд відомих підходів до паралельної реалізації показав, що основна ідея розпаралелювання полягає у одночасному виконанні незалежних операцій всередині одного шару мережі з подальшою синхронізацією між шарами. Окремо відзначено, що практичні реалізації можуть орієнтуватися як на багатоядерні CPU, так і на GPU-платформи, де додаткове значення мають особливості спільної пам'яті та накладні витрати на координацію [3, 4, 6].
